# Library 资产链路改造后端问题记录

日期：2026-03-06

## 背景

这份记录基于当前代码改动的静态 review。

当前后端基本处于不可验证状态，所以这里不把结论写成“运行时已复现”的口径，而是明确标注为“高置信度代码问题”。这些问题已经足够解释为什么现在后端即使接口存在，也很难被当成可用的服务端资产系统来验证。

## 当前结论

- 当前服务端资产链路还不能支持可靠验证。
- 主要阻塞点不在 UI，而在鉴权、跨端下行同步、去重后的删除语义、并发上传一致性，以及真实存储依赖。
- 如果不先修这些点，继续做联调会很难区分“功能没实现”和“后端本身不可信”。

## 问题列表

### P0: JWT 未验签，用户隔离实际上不成立

涉及文件：

- `api/assets/_auth.ts`
- `src/lib/authToken.ts`

现状：

- 服务端只是 base64 解码 Bearer token 的 payload，然后直接取 `sub` 当 `userId`。
- 客户端默认生成的是 `alg: none` 的开发 token。

影响：

- 任意请求方都可以伪造任意 `sub`，直接访问别人的资产数据。
- “按用户隔离”的前提当前并不存在。

建议：

- 接入真实 JWT 验签。
- 同时校验签名、issuer、audience、过期时间。
- 开发 token 只允许在明确的 dev/mock 模式下使用，不能默认混入正式接口逻辑。

### P1: `/api/assets/changes` 无法支撑新设备下行同步

涉及文件：

- `api/assets/changes.ts`
- `src/stores/assetStore.ts`

现状：

- `/api/assets/changes` 只返回 `remoteAssetId`、`contentHash`、`updatedAt`、`deletedAt`。
- 客户端对账逻辑只会更新“本地已经存在的资产”，不会创建缺失的本地资产。

影响：

- 新设备或空库首次进入时，即使服务端已有资产，也拉不下来。
- 现在的“服务端同步”更像是本地资产补状态，不是真正的跨端同步。

建议：

- `changes` 需要返回足够的资产元数据，或者补一个按 `remoteAssetId` 拉详情的接口。
- 客户端需要支持“本地不存在但远端存在”的下行建模。

### P1: 本地成功同步时会提前推进 cursor，可能漏掉别端变更

涉及文件：

- `src/stores/assetStore.ts`
- `api/assets/_store.ts`

现状：

- 本地上传/删除成功后，客户端会先推进 `lastRemoteChangeCursor`。
- 后端查询条件使用的是 `updated_at > since`。

影响：

- 如果其他设备的写入时间落在“旧 cursor”和“这次本地成功时间”之间，这个客户端后面就可能永远拿不到那条变更。
- 这是静默丢数据问题，最难排查。

建议：

- 不要用本地成功时间直接推进 cursor。
- cursor 应该只基于服务端返回结果推进，或者改成单调递增版本号/服务端水位。

### P1: 删除一个本地副本会删掉共享远端资产

涉及文件：

- `src/stores/assetStore.ts`
- `api/assets/_store.ts`

现状：

- 服务端按 `(userId, contentHash)` 去重，多条本地资产可能共享一个远端资产。
- 当前删除逻辑只要看到 `remoteAssetId` 就直接排远端删除任务。

影响：

- 删除一个本地副本，会把同 hash 的共享远端对象一起删掉。
- 其他仍存在的本地副本会变成悬挂引用。

建议：

- 本地删除远端资产前，先判断是否还有其他本地资产引用同一个 `remoteAssetId` 或同一个 `contentHash`。
- 更稳妥的方式是把“本地条目”和“远端资源”拆成两层模型，删除时做引用计数或引用关系判断。

### P2: 并发同 hash 上传时，客户端可能记住一个不存在的 `remoteAssetId`

涉及文件：

- `api/assets/_store.ts`
- `api/assets/complete-upload.ts`

现状：

- 服务端用 `(userId, contentHash)` 做 upsert 去重。
- 冲突时虽然会复用已存在记录，但没有把数据库最终保留的 canonical `assetId` 返回给客户端。

影响：

- 并发上传同 hash 文件时，某个客户端可能把本地状态记成自己的 `remoteAssetId`，但数据库里实际保留的是另一个 `assetId`。
- 后续 delete、change reconcile、查详情都会漂移。

建议：

- `complete-upload` 在落库后必须返回数据库最终确认的 canonical `assetId`。
- 客户端只以服务端最终返回值更新 `remote.remoteAssetId`。

### P2: 真实后端依赖未声明，运行时可能静默退回内存模式

涉及文件：

- `api/assets/_store.ts`
- `package.json`

现状：

- 代码里动态导入了 `pg`、`@aws-sdk/client-s3`、`@aws-sdk/s3-request-presigner`。
- 但当前 `package.json` 没有声明这些依赖。
- 初始化失败后会直接 fallback 到内存模式。

影响：

- 表面上接口能跑，实际上资产元数据和对象并没有真实持久化。
- 这会让联调阶段产生“接口没报错但数据不见了”的假象。

建议：

- 明确把真实后端依赖写进 `package.json`。
- 启动时打印当前使用的是 `memory` 还是 `postgres+s3`，不要静默降级。

## 为什么现在后端难以验证

核心原因不是单一 bug，而是以下几层一起失效：

1. 身份模型不可信，导致用户隔离无法验证。
2. 下行同步不完整，导致服务端有数据也拉不下来。
3. cursor 语义有漏洞，导致跨端变更会漏。
4. 去重后的删除模型不闭环，导致远端状态会被误删。
5. 真实存储依赖可能根本没启用，导致所有“同步成功”都只是内存假象。

## 最小修复顺序

1. 先修 JWT 验签和用户隔离。
2. 再补真实的服务端资产下行接口，确保新设备能拉全量或增量详情。
3. 修正 cursor 语义，只用服务端确认过的游标推进。
4. 修正去重后的删除语义，避免一个本地副本删掉共享远端资源。
5. 让 `complete-upload` 返回 canonical `assetId`，收敛并发去重一致性。
6. 明确声明并安装 Postgres/S3 依赖，同时暴露当前后端运行模式。

## 备注

如果后续继续推进这条链路，建议先把“服务端可验证”作为里程碑单独收口，再继续叠加 AI 保存、URL 导入、历史补传等能力。否则前面的基础不稳，后面的功能越多，定位问题越困难。
